# A/B 测试邮件营销活动

## 关键概念

| 概念 | 解释 |
| --- | --- |
| 转化率 | 完成目标行为的用户比例。本题中，目标行为是注册 UberEats。 |
| 二元事件 | 只有两种结果的事件，例如注册或不注册。 |
| 伯努利随机变量 | 以概率 `p` 取值为 1、以概率 `1 - p` 取值为 0 的随机变量。单个用户是否注册可以这样建模。 |
| 二项分布 | `n` 次独立伯努利试验中，成功次数所服从的分布。某一组的总注册人数可以这样建模。 |
| 中心极限定理 | 当样本量足够大时，样本均值或样本比例会近似服从正态分布，因此可以使用 z 检验。 |
| 零假设 | 默认假设，通常表示两组之间没有统计意义上的差异。 |
| p 值 | 在零假设为真时，观察到当前结果或更极端结果的概率。 |
| 统计显著性 | 在预先设定的阈值下，例如 `alpha = 0.05`，证据足够强，可以拒绝零假设。 |
| 护栏指标 | 在优化主指标时，用来确保其他重要体验或业务结果没有受损的指标。 |

## 问题

你在 Uber 的数据科学团队工作，想验证针对新功能 UberEats 的邮件营销活动是否能提升该功能的注册转化率。

假设你拥有非常大的样本。

请从统计学角度描述你会如何运行这个实验，并说明：

- 你会使用什么统计检验。
- 你会如何判断统计显著性。
- 这个实验在支持产品决策时可能有哪些局限。

## 解题思路

这是一道偏统计的 A/B 测试题。面试官不是只想听到“分成实验组和对照组”，而是想听到你是否理解这个实验背后的统计模型。

核心逻辑是：

1. 将符合条件的用户随机分到对照组和实验组。
2. 只给实验组发送邮件营销活动。
3. 在预先定义好的归因窗口内，观察每个用户是否注册 UberEats。
4. 将注册行为视为二元结果。
5. 用比例检验比较两组转化率。
6. 解释统计显著性和实际业务显著性。
7. 在给出产品建议前，讨论实验局限。

## 提示

### 提示 1：注册是二元事件

对每个用户来说，注册行为只有两种结果：

- `1`：用户注册了 UberEats。
- `0`：用户没有注册 UberEats。

因为实验会覆盖大量用户，所以我们可以把这些二元结果的平均值看作转化率。

### 提示 2：定义零假设

零假设通常表示组与组之间没有显著可观察变化。

在本题中：

- 零假设：邮件营销活动不会改变 UberEats 的注册转化率。
- 备择假设：邮件营销活动会提高 UberEats 的注册转化率。

用符号表示：

- `H0: p_control = p_treatment`
- `H1: p_treatment > p_control`

如果题目明确问的是是否“提高”转化率，一侧检验是合理的。如果团队关心任何变化，包括负面影响，则可以使用双侧检验。

### 提示 3：大样本很重要

统计检验有很多种，但样本量很大时，两比例 z 检验尤其合适。

因为注册是二元变量，每组注册人数服从二项分布。样本量足够大时，根据中心极限定理，样本转化率近似服从正态分布。

## 挑战

### 挑战 1

实验是否对目标变量有特定分布假设？

有。用户层面的注册行为可以建模为伯努利随机变量。一组用户中的总注册人数可以建模为二项分布。样本量很大时，样本转化率近似正态分布。

### 挑战 2

基于这些目标变量，你会运行哪种具体统计检验？

使用两比例 z 检验，比较对照组和实验组的注册转化率。

### 挑战 3

你是否说明了在这些实验假设下如何判断显著性？

应该计算 z 统计量，将其转换为 p 值，并把 p 值与预先选定的显著性水平比较，例如 `alpha = 0.05`。同时，也应该报告 treatment effect 的置信区间。

## 实验设计

### 1. 定义实验人群和资格条件

实验应在符合条件的 Uber 用户中运行。

资格条件必须在实验开始前定义清楚。例如：

- 位于 UberEats 可服务地区的现有 Uber 用户。
- 尚未注册 UberEats 的用户。
- 可以合法、合规接收营销邮件的用户。
- 没有选择退出促销邮件的用户。

这一点很重要，因为实验估计出的效果只适用于被纳入实验的人群。

### 2. 随机分组

创建两个组：

- 对照组：不接收 UberEats 邮件营销活动。
- 实验组：接收 UberEats 邮件营销活动。

随机化应在用户层面进行，这样两组在观察到和未观察到的特征上，平均来说都会比较均衡。

随机化后应检查的重要用户变量包括：

- 人口统计信息。
- 地理位置。
- 历史 Uber 使用情况。
- 如果有数据，也可以看历史外卖使用情况。
- 历史邮件打开或点击行为。
- 设备类型。
- 时区。

如果两组在这些变量上明显不均衡，估计出的实验效果可能会有偏。

### 3. 定义核心指标

主指标是 UberEats 注册转化率。

对每一组：

`转化率 = 注册用户数 / 被分配到该组的用户数`

注册归因窗口要在实验前定义。例如：

- 收到邮件后 7 天内注册。
- 收到邮件后 14 天内注册。
- 在一组邮件营销活动的第一封邮件后 30 天内注册。

归因窗口很重要，因为邮件营销的影响可能有延迟。

### 4. 定义假设

设：

- `p_A` 是对照组真实转化率。
- `p_B` 是实验组真实转化率。

如果题目关注邮件活动是否提高转化率：

- `H0: p_B = p_A`
- `H1: p_B > p_A`

如果团队想检测任何差异，不管正向还是负向：

- `H0: p_B = p_A`
- `H1: p_B != p_A`

面试中最好明确说明你选择哪一种，以及为什么。

## 统计模型

### 用户层面的注册行为

对每个用户 `i`，定义：

- `Y_i = 1`：用户注册 UberEats。
- `Y_i = 0`：用户没有注册。

则：

`Y_i ~ Bernoulli(p)`

这意味着每个用户的注册结果就像一次带权重的抛硬币，`p` 是注册概率。

### 组层面的注册人数

如果 A 组有 `n_A` 个用户，转化概率为 `p_A`，那么 A 组总注册人数服从：

`X_A ~ Binomial(n_A, p_A)`

同理：

`X_B ~ Binomial(n_B, p_B)`

观测到的转化率是：

- `p_hat_A = X_A / n_A`
- `p_hat_B = X_B / n_B`

### 为什么适合用 z 检验

因为样本量非常大，中心极限定理告诉我们，样本比例会近似服从正态分布。

因此，可以使用两比例 z 检验来比较 `p_hat_A` 和 `p_hat_B`。

## 统计检验

### 两比例 z 检验

使用两比例 z 检验比较两组转化率。

观测到的 treatment effect 是：

`delta_hat = p_hat_B - p_hat_A`

在零假设下，两组拥有相同的转化概率。此时使用合并转化率：

`p_hat_pooled = (X_A + X_B) / (n_A + n_B)`

零假设下的标准误是：

`SE = sqrt(p_hat_pooled * (1 - p_hat_pooled) * (1/n_A + 1/n_B))`

z 统计量是：

`z = (p_hat_B - p_hat_A) / SE`

然后从标准正态分布中计算 p 值。

### 判断统计显著性

实验前要先选择显著性水平，例如：

- `alpha = 0.05`，对应 95% 置信水平。
- 如果决策风险更高，或者比较次数很多，可以使用 `alpha = 0.01`。

如果使用一侧检验：

- 当 `p-value < alpha` 且 `p_hat_B > p_hat_A` 时，拒绝零假设。

如果使用双侧检验：

- 当 `p-value < alpha` 时，拒绝零假设，然后再看效果方向。

如果拒绝零假设，并且实验组转化率显著高于对照组，那么证据表明邮件营销活动提升了 UberEats 注册转化率。

## 实际业务显著性

统计显著还不够。

当样本量很大时，非常小的提升也可能达到统计显著。因此团队还要判断这个效果是否足够大、是否值得做。

有用的产品决策检查包括：

- 绝对提升：`p_hat_B - p_hat_A`。
- 相对提升：`(p_hat_B - p_hat_A) / p_hat_A`。
- 增量注册人数：由活动额外带来的注册数。
- 每个增量注册的营销成本。
- 新获得用户的下游价值。
- 提升幅度是否超过最小可检测效果或最小值得上线效果。

例如，从 10.00% 提升到 10.05%，在超大样本下可能统计显著，但未必值得承担工程、营销或品牌成本。

## 局限和注意事项

### 1. 注册不能反映退订

把注册建模为伯努利随机变量，可以回答用户是否注册。

但是它不能捕捉用户是否：

- 取消订阅邮件。
- 把邮件标记为垃圾邮件。
- 屏蔽未来沟通。
- 对 Uber 或 UberEats 产生负面情绪。

这些结果可能降低用户长期价值。

团队应把退订也作为另一个二元结果来追踪：

- `1`：用户退订。
- `0`：用户没有退订。

它可以用另一个两比例 z 检验来分析，并作为运行新营销活动的成本或护栏指标。

### 2. 邮件营销可能有延迟效果

答案不能忽略时间延迟。

邮件营销通常会随时间发生作用：

- 用户可能没有打开第一封邮件。
- 用户可能在第四封邮件后才转化。
- 用户可能收到邮件 20 天后才行动。
- 用户可能需要多次接触后才注册。

因此，实验必须定义足够长的观察窗口。团队也可以用 cohort 曲线分析随时间变化的转化情况。

### 3. 转化提升不等于产品成功

即使邮件活动提高了注册转化率，也不代表一定应该上线。

可能的问题包括：

- 更高的退款率或取消率。
- 更低的首单完成率。
- 邮件拉来的用户留存更差。
- 如果活动吸引的是低意向用户，用户终身价值可能更低。
- 如果邮件太激进，用户可能反感。

A/B 测试结果必须放到业务和产品语境中解读。

### 4. 干扰和污染

干净的统计检验假设对照组和实验组用户互不影响。

这个假设可能在以下情况中被破坏：

- 实验组用户把邮件转发给对照组用户。
- 用户住在同一家庭并互相影响。
- 活动触发了社交传播或推荐效应。
- 用户在其他营销渠道中看到了同一活动。

如果干扰很可能存在，实验可能需要按 cluster 或地理区域设计。

### 5. 多重比较

如果团队同时测试很多邮件标题、用户分层、时间窗口和指标，假阳性概率会上升。

处理方式包括：

- 预先注册主指标和假设。
- 控制探索性分析范围。
- 如果要做很多正式比较，需要进行多重比较校正。
- 如果 segment 分析没有提前设计并保证统计功效，只把它当作方向性发现。

### 6. 外部有效性

实验结果只适用于被测试的用户和活动条件。

它未必能推广到：

- 其他城市。
- 其他季节。
- 不同邮件素材。
- 未来用户 cohort。
- 已选择退出营销邮件的用户。
- UberEats 可用性或餐厅密度不同的市场。

## 决策框架

| 结果 | 产品决策 |
| --- | --- |
| 统计显著且业务上有意义，护栏指标健康 | 上线或扩大营销活动。 |
| 统计显著但提升很小 | 结合成本、长期价值和运营复杂度决定是否值得做。 |
| 没有显著提升 | 不要大规模上线；迭代邮件素材、目标人群或渠道。 |
| 转化提升但退订、垃圾举报或留存变差 | 不应按当前形式上线；需要重新设计活动或缩小目标人群。 |
| 只在特定 segment 有提升 | 如果该 segment 是预先定义且样本量足够，可以考虑定向发布。 |

## 30 秒回答

我会将符合条件的 Uber 用户随机分成对照组和实验组，只给实验组发送 UberEats 邮件营销活动。主指标是在预先定义的归因窗口内是否注册 UberEats。由于注册是二元事件，单个用户结果可以建模为伯努利随机变量，一组注册人数服从二项分布；样本量很大时，转化率根据中心极限定理近似正态。因此我会用两比例 z 检验检验 `H0: p_control = p_treatment`，计算 z 统计量和 p 值，并在预先设定的显著性水平如 0.05 下判断是否拒绝零假设。如果实验组转化率显著更高，还要继续检查实际提升幅度、退订率、垃圾举报、留存、用户终身价值和延迟转化效果，再决定是否上线。
